## Problem

Threat hunting work often gets split across reports, notes, KQL scratchpads, cloud shells, and after-action documents. Hunter-7 addresses the coordination problem: how to turn a hunt idea into a reviewed plan, executable analytics, and durable findings without hiding the analyst's reasoning.

## Constraints

- CAC-authenticated and cloud-shell execution paths had to remain compatible with government environments.
- Live telemetry execution required review gates before touching Sentinel or mission systems.
- Public documentation needed to explain the architecture without exposing private targets, tenant details, or operational secrets.

## Approach

Hunter-7 packages a hunt as a protocol. Skills cover research, hunt focus, ATT&CK mapping, data-source selection, KQL or SQL analytic generation, blueprint assembly, and findings writing. A GitLab Pages control plane presents graph, MCP resource, workflow, static search, and generated knowledge artifacts for review.

## Artifact And Proof

The public artifact is a sanitized architecture note that shows the workflow boundaries: report ingestion, knowledge indexing, staged skills, Sentinel MCP configuration, and protocol run folders.

## Outcomes And Status

Hunter-7 is best framed as an agentic investigation system rather than a single script. The durable value is the operating model: structured hunt workspaces, review gates, generated knowledge artifacts, and clear handoffs between planning and execution.
